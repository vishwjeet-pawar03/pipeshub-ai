## 脆弱性の報告

私たちはセキュリティ脆弱性を重く受け止めており、発見した問題を責任を持って開示しようとするあなたの取り組みに感謝します。

### 報告方法

**セキュリティ脆弱性について、公開の GitHub issue を作成しないでください。**

代わりに、次のアドレスにメールでセキュリティ脆弱性を報告してください：**abhishek@pipeshub.com**

### 含めるべき内容

脆弱性を報告する際は、次の内容を含めてください：

- 脆弱性の説明
- 問題を再現する手順
- 潜在的な影響の評価
- 修正案（あれば）
- フォローアップ用のあなたの連絡先

### 期待できること

- **初回応答**：48 時間以内に報告の受領を確認します
- **評価**：5 営業日以内に初期評価を提供します
- **更新**：解決まで 7 日ごとに進捗状況をお知らせします
- **解決**：重大な脆弱性は 30 日以内の解決を目指します
- **クレジット**：セキュリティアドバイザリであなたの貢献に謝意を示します（匿名を希望される場合を除く）

### 開示のタイムライン

- **0 日目**：脆弱性が報告される
- **1〜2 日目**：初回の確認
- **1〜5 日目**：初期トリアージと影響評価
- **1〜30 日目**：修正の開発とテスト
- **30 日目以降**：修正が展開された後に公開開示、通常は初回報告から 90 日以内。

### 脆弱性の評価

私たちは次の基準を用いて脆弱性を分類します：

- **クリティカル（Critical）**：ユーザーデータまたはシステムの完全性に対する即時の脅威
- **高（High）**：広範なユーザーへの影響を伴う重大なセキュリティ影響
- **中（Medium）**：限定的な影響範囲を伴う中程度のセキュリティ影響
- **低（Low）**：影響が最小限の軽微なセキュリティ問題

### セキュリティのベストプラクティス

本プロジェクトに貢献する際は：

- 依存関係を最新の状態に保つ
- 安全なコーディング慣行に従う
- すべてのユーザー入力を検証する
- 適切な認証と認可を使用する
- セキュリティイベントのログ記録を実装する
- 定期的なセキュリティテストを推奨します

### 適用範囲

本セキュリティポリシーは次に適用されます：

- メインのアプリケーションコード
- 公式の Docker イメージ
- セキュリティに影響を与える可能性のあるドキュメント
- 私たちが直接保守する依存関係

### 適用範囲外

次の事項は通常、セキュリティ脆弱性とはみなされません：

- サードパーティの依存関係に関する問題（該当する保守担当者に報告してください）
- ソーシャルエンジニアリング攻撃
- 物理的セキュリティの問題
- 増幅を伴わないリソース枯渇によるサービス妨害

### セーフハーバー

私たちは、次の条件を満たすセキュリティ研究者にセーフハーバーを提供します：

- プライバシー侵害やサービス中断を回避するよう誠実に努める
- 自身が所有するアカウント、または明示的な許可があるアカウントとのみやり取りする
- 他のユーザーのデータにアクセスまたは変更しない
- 脆弱性を速やかに報告する
- 解決前に脆弱性を公開開示しない

### 連絡先

本セキュリティポリシーに関するご質問は、次までご連絡ください：security@pipeshub.com
